# ADR-04.006: Use-Case Factories Use a Single Explicit Signature

## Status

Accepted at 2026-06-21

## Context

Use-case factories are one of the most repeated architectural entry points in
the backend. When their shape varies too much, imports become inconsistent,
composition code becomes harder to scan, and architecture checks lose a stable
target.

We already require use-case dependency contracts to remain private. The next
step is to keep the factory boundary itself equally explicit and predictable.

## Decision

- Every use-case implementation must export a named factory.
- Factory names must end with `Factory`.
- Factories must accept exactly one dependency parameter.
- That parameter must be typed with a contract whose name ends with
  `Dependencies`.
- Default exports are forbidden for use-case factories.

The parameter may be destructured, but the factory boundary must remain a single
dependency contract.

## Consequences

- Composition roots can wire use-cases in a uniform way.
- Factory naming becomes easy to scan and grep.
- Architectural checks can detect accidental drift quickly.
- The rule is slightly stricter than free-form JavaScript style, but the
  trade-off is acceptable because it improves consistency and toolability.
