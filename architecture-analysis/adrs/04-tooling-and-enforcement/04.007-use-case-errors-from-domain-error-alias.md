# ADR-04.007: Use-Case Errors Must Flow Through the Domain Error Alias

## Status

Accepted at 2026-06-21

## Context

Use-cases should expose a single, stable error source so the domain layer stays
consistent and easy to audit. Allowing direct imports from internal error
modules creates drift in import style and makes the boundary harder to scan.

The backend already exposes a canonical alias for domain errors. Use-cases
should consume that alias instead of importing error classes from lower-level
paths.

## Decision

- Use-case implementation, support, and test files must import error classes
  from `@DomainError`.
- Direct imports from `@GenericSubdomains/utils/domainError`,
  `@GenericSubdomains/utils/domain-error/domainError`, or any other
  non-`@DomainError` error module are forbidden.
- The rule applies to use-case files, including policy helpers and tests under
  `use-cases/`.

## Consequences

- Use-case error imports become consistent across the codebase.
- The canonical domain error entry point stays obvious.
- Architecture checks can detect and report inconsistent imports automatically.
